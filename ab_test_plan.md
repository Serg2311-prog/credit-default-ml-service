# A/B Test Plan: Credit Default Model (v1 vs v2)

## 1. Цель

Сравнить текущую модель `v1` (control) и новую модель `v2` (treatment) на онлайн-трафике,
чтобы подтвердить улучшение качества обнаружения дефолтных клиентов без деградации бизнес-рисков.

## 2. Гипотезы

- **H0 (нулевая):** качество `v2` не лучше `v1` по целевым метрикам.
- **H1 (альтернативная):** `v2` показывает статистически значимое улучшение относительно `v1`.

## 3. Дизайн эксперимента

- Распределение трафика: **50/50**.
- Группы:
  - **A:** модель `v1`
  - **B:** модель `v2`
- Рандомизация: по `customer_id` (стабильный хеш для консистентного роутинга).
- Единица наблюдения: одно кредитное решение.
- Продолжительность: до достижения статистической мощности и минимального объема наблюдений.

## 4. Метрики

Основные:
1. **F1-score** (баланс precision/recall)
2. **Recall** для класса default=1 (приоритетно с точки зрения риска)

Дополнительные guardrail:
- доля отклоненных заявок,
- latency API,
- error rate API.

## 5. Статистический анализ

### 5.1 z-test

Для бинарных исходов (например, recall@decision по итогам появления ground truth):
- сравнение долей между группами A/B с использованием двухвыборочного z-теста для пропорций.

### 5.2 Доверительные интервалы

- Строим **95% CI** для разницы метрик `(metric_B - metric_A)`.
- Для F1 можно использовать bootstrap CI (рекомендуется из-за сложной формы метрики).

## 6. Критерий успеха

Эксперимент считается успешным, если одновременно выполнено:
1. `Recall_B - Recall_A > 0` и 95% CI не пересекает 0.
2. `F1_B - F1_A > 0` и 95% CI не пересекает 0.
3. Нет деградации guardrail-метрик выше допустимого порога:
   - API error rate не растет более чем на 0.2 п.п.
   - P95 latency не растет более чем на 10%.

## 7. Риски и контроль качества

- **Data drift:** мониторинг распределений входных признаков.
- **Label delay:** учитывать задержку в появлении факта дефолта.
- **Selection bias:** стабильный роутинг по customer_id.
- **Operational incidents:** fallback на `v1` при аномалиях.

## 8. Rollout plan

1. Shadow mode для `v2` (без влияния на решение).
2. A/B 50/50 в проде.
3. Анализ статистики.
4. При успехе: staged rollout 10% → 50% → 100%.
5. При провале: rollback на `v1`, разбор причин, повторное обучение.
